# Active intent mode invariant

Each controlled snake in centaur play has exactly ONE active next-move source at a time, modeled as a single discriminated union `SnakeIntent` on `ControlledSnake.intent` (in `src/server/active-game-manager.ts`): `{kind:'heuristic'} | {kind:'manual', move} | {kind:'queue', cells} | {kind:'goto', targets: Coord[]} | {kind:'near', target}`. `IntentMode = SnakeIntent['kind']`. The union IS the mutual exclusion — there are no separate `premoveQueue`/`waypoint`/`activeIntentMode`/`userSelectedMove` fields to keep in sync.

**Goto/near are WEIGHTED VOTES in the heuristic matrix, not hard overrides (2026-07 redesign).** The intent stores ONLY the durable targets (`goto` a QUEUE of targets — `targets[0]` active, shifted on arrival, Shift+Alt+click appends; `near` a single cell). Everything else is derived per turn: (a) the evaluator scores a per-move `gotoProgress`/`nearProgress` stat (+1 first step of a BFS shortest path, 0 sideways, −1 backward, −2 cut off / landing on a near target), weight = config `gotoProgress`/`nearProgress` (default 300/250 — above food/territory, below deaths −500/trapped −600, so survival always outvotes the target); (b) at staging, `getWaypointBiasedMove` re-derives the choice from this turn's `moveEvaluations` (subtract the recorded waypoint contribution, add the CURRENT target's `weight × stat`, then the shared `pickBestMove` = trapped-veto + argmax, exported by decision-engine) so a waypoint set/moved mid-turn takes effect immediately and the staged move is always "best matrix output with the waypoint integrated"; (c) the rendered green route lives on `ControlledSnake.gotoRoute` (a DERIVED display cache, NOT in the intent), recomputed by `refreshGotoRoute` on every stage and commit as `[stagedDestination, ...shortestPath(stagedDestination → targets[0])]` while a move is staged, or the plain shortest path from the projected head after commit. The shared pathfinder + stat functions live in `src/logic/waypoint-pathing.ts` — evaluator, re-bias, and display all use it, so visual/stat/move can never diverge. Track the two paths separately: the optimal-from-current-head analysis feeds the matrix; the staged-conditioned path is what renders.

**Rule:** all intent changes go through the single helper `setIntent(gameId, snakeId, intent)`. It assigns `controlled.intent` whole (replacing the union structurally supersedes the prior mode's state — no field-by-field clearing) and then calls `stageMove`. Logs a one-line warning when a still-staged manual is replaced by a non-manual intent (so a manual move never silently disappears mid-turn). Never assign `controlled.intent` directly elsewhere. Queue cells / goto targets are mutated in place only by their owning advance helpers (`advancePremoveQueueAfterMove` shifts `intent.cells`; the goto arrival check in `updateGameState` shifts `intent.targets`; `setWaypoint(append)` toggles a target's queue membership), which then call `stageMove`/`setIntent` themselves.

**Client/WS contract is a PROJECTION of the union.** The wire carries `premoves`, `waypoints`, `routes`, `activeIntentModes`, `stagedMove` as derived getters over `intent` (e.g. `getPremovesForGame` returns `intent.cells` only for `kind==='queue'`). `waypoints` is `{[snakeId]: {type:'green'|'blue', cells: Coord[]}}` — green carries the whole goto queue in order; `activeIntentModes` values include `'goto'`/`'near'`. The client (`play-game.html`) mirrors these into its own locals. Update both ends in lockstep when changing the projection shape.

**Selection vs. staging vs. commit (the three-concept separation).** These are distinct and must not be conflated:
- SELECTION = the client-side purple-box inspection cursor only (candidate click / arrows / WASD). It updates the breakdown table + territory overlay and NOTHING else: no server message, no staged-move write, no queue clear. (`selectMove` in `play-game.html`.)
- STAGING = the single stored `staged: StagedMove | null` per snake (server-owned, on `ControlledSnake`), broadcast via the `stagedMoves` wire map (grey = heuristic/bot, user color = human). The client stages only via Space / Stage button → `stageSelectedMove` → `select-move` ws message → `setUserSelection` (manual mode, drops queue/waypoint). Queue/waypoint modes stage server-side through `stageMove`.
- **Broadcast colour/source must derive from the TRUE resolved origin, not the nominal mode.** `stageMove` binds the `IntendedMove.source` from `computeIntendedMove` into `staged.source`. The staged-broadcast colours grey iff `staged.source ∈ {bot, fallback}` — NOT iff `intent.kind === 'heuristic'`. **Why:** a waypoint/queue can fall back to the bot's recommendation for a turn; if you colour by mode it renders in the user's colour while actually committing the bot's move, so a user-coloured arrow no longer guarantees the user's move (this was "Bug A": the arrow stayed my colour but a bot decision came in). The `computeIntendedMove` bot-fallback branch must therefore return `source: 'bot'` (it used to return `'waypoint'` whenever a waypoint was set — the disguise). Silent fallbacks are logged in `logStagedMoveAnomalies` (called by `stageMove`); mid-turn replacements of a still-staged manual selection are logged in `setIntent`.

- **Reactive staged-arrow sync.** `stageMove` (the choke point every intent change funnels through) ends by calling the coalesced `notifyStagedChange(gameId)` — it marks the game dirty and schedules ONE `setImmediate().unref()` flush per tick that fires `onStagedChange(gameId)` → `broadcastSelectionsUpdate`. **Why:** previously every WS handler (`select-move`/`set-premove`/`set-waypoint`) broadcast explicitly, missing auto-pilot/server-side stage changes and duplicating broadcasts. With the reactive path, those handlers no longer broadcast themselves. `.unref()` so a pending flush never keeps the process alive.
- COMMIT = ONLY the per-snake safety timer at the turn deadline, which reads the stored `staged` record (via `commitStagedMove`/`stagedMoveForTurn`). There is exactly ONE commit path. **Do not re-introduce alternate non-deadline commit paths** (explicit client submit, client auto-commit timers, unselected-snake auto-pilot, hold-release commit, first-turn auto-pilot). The armed-suicide kill is the one intentional exception.

**The deadline-timer commit path has NO surrounding try/catch — a throw there crashes the whole server.** Anything `resolvePendingMove` calls at commit time (e.g. the board-safety validation `ensureBoardSafeMove`, which builds a `BoardGraph`) MUST be exception-safe and swallow its own errors, returning the move unchanged on failure. Separately, the committed move is validated against the current board before sending (`isPassableAtTurn(dest, 1)` — turn-1 semantics so a vacating tail stays valid) and replaced with a passable / in-bounds direction if the staged move (or a stale bot fallback) would step off-board. Suicide is the one exception (it intentionally steers into death, so it bypasses validation).

**Because the deadline commit is a pure READ of the stored `staged` record, refresh-trigger coverage is a correctness requirement, not an optimization.** Every input that changes a snake's intended move MUST call `stageMove`, or the timer silently commits a stale direction. The easy-to-miss ones: (a) per-snake `/move` completion must re-stage for ALL non-manual modes, not just `heuristic` — a `waypoint` snake that isn't the turn-advancer updates its `gotoRoute` but never hits the turn-rollover refresh pass, so without a per-snake re-stage its staged move stays a turn stale; (b) hold-release/hold must re-stage after clearing a manual selection. Manual is the only mode safe to skip on bot completion (its move was set explicitly and is single-turn; `computeIntendedMove` precedence keeps a late bot result from overriding any human mode anyway).

**Staged-arrow broadcast must not gate on an in-flight `pendingMove`.** The staged map is derived from each snake's stored `staged` record; gate only on that. The board-update broadcast fires once per turn from the single snake that advanced it, so the per-snake `snake-turn-update` (and `move-committed`) broadcasts must also carry the staged map, or every other snake's grey arrow goes missing until something unrelated rebroadcasts. The client only draws arrows for snakes present on the board, so eliminated snakes are skipped there.

**Why:** a prior design had ~5 commit paths plus client-side optimistic commits and a purple checkmark overlay; staged state could diverge from what actually committed. Collapsing to "server owns `staged`; deadline timer is the sole committer" makes the staged arrow a guaranteed-accurate preview of the commit.

**Route/queue anchoring:** the goto route and the premove queue must both anchor at the *projected head* (the cell the snake occupies after any move already committed this turn — `getProjectedHead`), not the live head. After committing a move, recompute the goto route from the projected head and re-broadcast it (e.g. on the move-committed message), or the rendered path and its first step start at the stale head. When nothing is committed, projected head == live head, so the same code path serves the not-yet-committed case.

**`computeIntendedMove` is the single move-decision chokepoint** — it is invoked through `stageMove`, which binds the resolved direction into the stored `staged` record. The deadline safety timer and previous-turn cleanup then just READ `staged` (via `commitStagedMove`); they no longer call `computeIntendedMove` directly. Priority: manual → queue → goto/near biased-matrix pick → bot → fallback. Manual and queue HARD-OVERRIDE the move (return their own direction). Goto/near return the OUTPUT OF THEIR OWN COMPUTATION (`getWaypointBiasedMove`, source `'waypoint'`) — a genuine re-scored matrix selection, never a relabel of `botRecommendation`; when this turn's evaluations are unavailable (turn 0, errors) they fall through and the move is truthfully labeled `'bot'`. The past "relabel" bug (waypoint branch returned `botRecommendation` tagged `'waypoint'`, so the snake never followed the green route it displayed) is now impossible by construction: the green route is DERIVED FROM the staged move (`refreshGotoRoute` puts the staged destination as `gotoRoute[0]`), so the visual follows the decision rather than a second mechanism competing with it. All anchoring still uses the PROJECTED head (matching `getPremoveDirection`).

**Why:** without one mutation point, multiple sources (queue + waypoint + manual) could be populated simultaneously and the source-priority resolver would pick inconsistently with what the UI shows.

**How to apply / edge cases:**
- Manual is a SINGLE-turn intent — it auto-reverts to `heuristic` on the snake's OWN `/move` in `setBotRecommendation`. Stale-vs-fresh is decided by the staged record's turn: revert iff `intent.kind==='manual' && staged.turn !== game.boardStateTurn` (a manual staged for an earlier turn is stale; one staged *this* turn during the bot's compute window stays authoritative and is re-derived). **Why:** the old check used `pendingHasManualSelection` (manual selection lived on the recreated-each-turn pending); now manual lives durably on `intent`, so the turn carried by the staged record is the freshness signal. Queue, goto and near PERSIST across turns.
- Manual staging clears the queue by design (manual mode), so a draining queue must NOT route through the manual path. Queue/goto advancement and staging happen server-side (`stageMove` / `advancePremoveQueueAfterMove` / the goto arrival check); only Space / Stage button → `select-move` → `setUserSelection` enters manual mode. There is no client-side auto-commit anymore.
- Goto arrival check in `updateGameState` (head or body[1] on `targets[0]`) SHIFTS the target queue; only an emptied queue reverts to `heuristic` via `setIntent`. Blue near never auto-clears (and the near stat makes the snake never step onto it).
- The green display route is OWNED by the server on `ControlledSnake.gotoRoute` (a derived cache OUTSIDE the intent — targets are the only durable state). It is recomputed ONLY by `refreshGotoRoute`, called from `stageMove` (after the staged record is bound — the route's first cell IS the staged destination) and from `resolvePendingMove` (post-commit re-anchor at the projected head). The strategy (`getBestMoveWithDebug`) must NOT return a route ("Bug B" history: a live-head-anchored strategy route desynced from the projected head). Because `setIntent`/`setWaypoint` funnel through `stageMove`, the path renders the instant the user alt-clicks. When computing off the shared `game.boardState`, go through `viewFor` (re-points `you`) because BoardGraph keys invulnerability/severability off `gameState.you`.
- The safety-timer commit buffer in `setPendingMove` is derived from the measured game-server round-trip ping (`gameServerPing + 30ms` jitter, floored at 60ms), NOT a flat 100ms — the committed response still has to travel back to the game server before `serverExpiryTime`, and on a slow link a flat buffer lets it land after the deadline (game server then applies its own "continue straight" default). Turn 0 keeps the large 5000ms warm-up buffer.

**A staged move is one atomic `StagedMove` product type `{ readonly snakeId, turn, move, source }`, nullable as a whole — never three structurally-separable fields, never a bare mutable direction; nothing about snake X's `/move` may touch snake Y's staged state.** The record is WRITTEN only by `stageMove` (the choke point, replacing the whole `controlled.staged`), and READ for commit only through `stagedMoveForTurn(controlled, snakeId, turn)`, which honours it solely when both snake AND turn align (rendering reads `cs.staged?.move/.source` without turn-gating). `readonly` fields force whole-value replacement so a partial update can't desync the tuple. **Why:** in multi-snake centaur, one snake's board-advancing `/move` used to re-derive (and stale-manual-revert) EVERY snake's staged move, so a snake whose prior-turn pending hadn't committed got its value clobbered with a new-turn move → it committed the wrong turn's move (180° into its own neck). A prior fix used three separate fields (`stagedMove`/`stagedMoveSource`/`stagedMoveTurn`) — rejected by the user as desync-prone and over-commented; the single product type is the canonical design. **How to apply:** (1) drop (`controlled.staged = null`) + re-stage a snake ONLY on its OWN `/move`, after that handler's previous-turn-cleanup has consumed the prior value; (2) the board-advance loop may reset shared per-turn bookkeeping (commit flags, hold counter) but must NOT call `stageMove`/`setIntent` on other snakes; (3) stale single-turn manual reverts to heuristic per-snake, detected by comparing the staged record's turn to the current board turn (`intent.kind==='manual' && staged.turn !== game.boardStateTurn`; a manual staged *this* turn during the bot's compute window is kept); (4) every commit site validates via `stagedMoveForTurn` and falls back to the bot move on mismatch as a backstop, not as the primary mechanism.

**Queue teardown must tolerate one ambiguous turn.** Never clear the premove queue just because its head can't resolve to an adjacent direction for a single turn (transient broadcast race / momentary non-adjacency). `advancePremoveQueueAfterMove` runs AFTER the move is committed, so it measures from the projected head (= the snake's real resulting cell) and does a three-way decision: DRAIN (projected == queue[0], pop it), HOLD (projected adjacent to queue[0] — bot covered one turn, keep queue + keep `queue` mode, resumes next turn), CLEAR (projected neither queue[0] nor adjacent — provably off the path = true divergence). **Why:** the old code pre-cleared the queue in the unselected-snake auto-pilot path whenever `getPremoveDirection` returned null, so one transient turn permanently silently deactivated the plan. There is NO infinite-hold risk: HOLD only happens when projected is adjacent to queue[0], which guarantees next turn's live head resolves the queue. During a held turn `intent.kind` intentionally stays `queue` while the move resolves as `bot` — that's expected, not an invariant violation.
