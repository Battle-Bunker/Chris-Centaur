# Active intent mode invariant

Each controlled snake in centaur play has exactly ONE active next-move source at a time: `activeIntentMode` ∈ `heuristic | manual | queue | waypoint` (on `ControlledSnake` in `src/server/active-game-manager.ts`).

**Rule:** all mode changes must go through the single helper `transitionIntentMode(gameId, snakeId, controlled, mode)`. It only CLEARS the other three sources' state (premoveQueue, waypoint+gotoRoute, this-turn manual selection) and then calls `refreshStagedMove` to recompute the stored staged move; callers populate the new mode's own state BEFORE the call (e.g. `setWaypoint` computes the gotoRoute first). Never set `activeIntentMode` directly elsewhere.

**Selection vs. staging vs. commit (the three-concept separation).** These are distinct and must not be conflated:
- SELECTION = the client-side purple-box inspection cursor only (candidate click / arrows / WASD). It updates the breakdown table + territory overlay and NOTHING else: no server message, no staged-move write, no queue clear. (`selectMove` in `play-game.html`.)
- STAGING = the single stored `stagedMove: Direction` per snake (server-owned, on `ControlledSnake`), broadcast via the `stagedMoves` map (grey = heuristic/bot, user color = human). The client stages only via Space / Stage button → `stageSelectedMove` → `select-move` ws message → `setUserSelection` (manual mode, drops queue/waypoint). Queue/waypoint modes stage server-side through `refreshStagedMove`.
- COMMIT = ONLY the per-snake safety timer at the turn deadline, which reads the stored `stagedMove`. There is exactly ONE commit path. **Do not re-introduce alternate non-deadline commit paths** (explicit client submit, client auto-commit timers, unselected-snake auto-pilot, hold-release commit, first-turn auto-pilot). The armed-suicide kill is the one intentional exception.

**Because the deadline commit is a pure READ of stored `stagedMove`, refresh-trigger coverage is a correctness requirement, not an optimization.** Every input that changes a snake's intended move MUST call `refreshStagedMove`, or the timer silently commits a stale direction. The easy-to-miss ones: (a) per-snake `/move` completion must refresh for ALL non-manual modes, not just `heuristic` — a `waypoint` snake that isn't the turn-advancer updates its `gotoRoute` but never hits the turn-rollover refresh pass, so without a per-snake refresh its staged move stays a turn stale; (b) hold-release/hold must refresh after clearing a manual selection. Manual is the only mode safe to skip on bot completion (its move was set explicitly and is single-turn; `computeIntendedMove` precedence keeps a late bot result from overriding any human mode anyway).

**Staged-arrow broadcast must not gate on an in-flight `pendingMove`.** The staged map is derived from each snake's stored `stagedMove`; gate only on that. The board-update broadcast fires once per turn from the single snake that advanced it, so the per-snake `snake-turn-update` (and `move-committed`) broadcasts must also carry the staged map, or every other snake's grey arrow goes missing until something unrelated rebroadcasts. The client only draws arrows for snakes present on the board, so eliminated snakes are skipped there.

**Why:** a prior design had ~5 commit paths plus client-side optimistic commits and a purple checkmark overlay; staged state could diverge from what actually committed. Collapsing to "server owns `stagedMove`; deadline timer is the sole committer" makes the staged arrow a guaranteed-accurate preview of the commit.

**Route/queue anchoring:** the goto route and the premove queue must both anchor at the *projected head* (the cell the snake occupies after any move already committed this turn — `getProjectedHead`), not the live head. After committing a move, recompute the goto route from the projected head and re-broadcast it (e.g. on the move-committed message), or the rendered path and its first step start at the stale head. When nothing is committed, projected head == live head, so the same code path serves the not-yet-committed case.

**`computeIntendedMove` is the single move-decision chokepoint** — it is invoked through `refreshStagedMove`, which writes the resolved direction into the stored `stagedMove`. The deadline safety timer and previous-turn cleanup then just READ `stagedMove`; they no longer call `computeIntendedMove` directly. Priority: manual → queue → goto-route-head → bot → fallback. Each non-bot mode must HARD-OVERRIDE the move (return its own direction), not merely relabel the bot's recommendation's `source`. A past bug: the waypoint branch returned `botRecommendation` tagged `'waypoint'`, so the snake never followed the green route it displayed — visual and move were two different mechanisms. Fix: `getGotoRouteDirection` returns `directionFromTo(head, gotoRoute[0])` and that drives the move.

**Why:** without one mutation point, multiple sources (queue + waypoint + manual) could be populated simultaneously and the source-priority resolver would pick inconsistently with what the UI shows.

**How to apply / edge cases:**
- Manual is a SINGLE-turn intent — it auto-reverts to `heuristic` on each new turn in `setBotRecommendation`'s board-updated loop (pendingMove is recreated null each turn). Queue and waypoint PERSIST across turns.
- Manual staging clears the queue by design (manual mode), so a draining queue must NOT route through the manual path. Queue/waypoint advancement and staging happen server-side (`refreshStagedMove` / `advancePremoveQueueAfterMove`); only Space / Stage button → `select-move` → `setUserSelection` enters manual mode. There is no client-side auto-commit anymore.
- Green ("goto") waypoint auto-clear in `updateGameState` must also clear `gotoRoute` and transition back to `heuristic`.
- `gotoRoute` (live green path) is recomputed every `/move` via `BoardEvaluator.computeWaypointRoute` (BFS, only for green waypoints) and stored on the snake in `setBotRecommendation`. It arrives `[]` when no reachable green waypoint, so it self-clears.
- It's ALSO computed synchronously in `setWaypoint` (via `computeGotoRouteNow`) so the path renders the instant the user alt-clicks, not only after the next `/move`. When computing off the shared `game.boardState`, build a `{...boardState, you: targetSnake}` because BoardGraph keys invulnerability/severability off `gameState.you`, and `boardState.you` is whichever snake last sent `/move`.
- Both green and blue waypoints map to `waypoint` mode; only green produces a rendered route.

**Queue teardown must tolerate one ambiguous turn.** Never clear the premove queue just because its head can't resolve to an adjacent direction for a single turn (transient broadcast race / momentary non-adjacency). `advancePremoveQueueAfterMove` runs AFTER the move is committed, so it measures from the projected head (= the snake's real resulting cell) and does a three-way decision: DRAIN (projected == queue[0], pop it), HOLD (projected adjacent to queue[0] — bot covered one turn, keep queue + keep `queue` mode, resumes next turn), CLEAR (projected neither queue[0] nor adjacent — provably off the path = true divergence). **Why:** the old code pre-cleared the queue in the unselected-snake auto-pilot path whenever `getPremoveDirection` returned null, so one transient turn permanently silently deactivated the plan. There is NO infinite-hold risk: HOLD only happens when projected is adjacent to queue[0], which guarantees next turn's live head resolves the queue. During a held turn `activeIntentMode` intentionally stays `queue` while the move resolves as `bot` — that's expected, not an invariant violation.
